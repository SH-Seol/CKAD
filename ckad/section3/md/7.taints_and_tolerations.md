# Taints and Tolerations

특정 노드에는 특정 앱만 띄우고 싶다거나 특정 노드는 점검 중이니 
파드가 오지 못하게 막고 싶다는 상황을 마주할 때 쓰이는 개념이 바로
Taints와 Toleration이다.

---

## 1. 기본 개념: 밀어내고 견디기

- Taint: 노드(사람)에 뿌리는 파드(벌레) 기피제이다.
- Toleration: 파드(벌레)가 가진 면역력이다.

핵심은 이들은 특정 노드가 파드를 거부하는 것이지 특정 파드를 특정 노드로 보내는 기능이 아니다.

---

## 2. Taint 설정하기
노드에 taint를 거는 명령어는 아래와 같다.
```bash
kubectl taint node <노드이름> key=value:taint-effect

# 예시
kubectl taint nodes node01 spray=mortein:NoSchedule

#taint 해제 예시
kubectl taint nodes controlplane node-role.kubernetes.io/control-plane:NoSchedule-
```

3가지 taint의 효과(effect)는 아래와 같다.

- `NoSchedule`: Toleration이 없으면 새로운 파드가 이 노드에 절대 스케줄링 되지 않음
- `PreferNoSchedule`: 가급적 피하지만, 클러스터에 자원이 아예 없으면 어쩔 수 없이 배치될 수도 있음
- `NoExecute`: 새로운 파드 생성 차단 및 이미 노드에서 실행 중인 파드 중 Toleration이 없는 파드 Evict

---

## 3. Toleration 설정하기

taint가 걸린 노드에 들어가야 하는 파드는 spec의 하위에 toleration 설정을 추가해야 한다.

```yaml
apiVersion: v1
kind: Pod
metadata:
	name: myapp-pod
spec:
	containers:
		- name: nginx-container
			image: nginx
	tolerations:
		- key: "app"
			operator: "Equal" #taint의 key/value와 정확히 일치해야함
			value: "blue"
			effect: "NoScedule"
```

---

## 4. Master Node의 비밀

k8s를 설치하면 마스터 노드에는 자동으로 taint가 걸린다. 그래서 우리가 `kubectl run` 으로 파드를 만들면 항상 워커 노드에만 뜨는 것이다.

- 확인 방법: `kubectl describe node <마스터노드명>` 명령어로 `Taints` 항목 확인
- 보통 `node-role.kubernetes.io/control-plane:NoSchedule` 같은 taint가 걸려 있어 일반 앱 파드가 마스터 노드에 못 들어간다.

---

#### 주의 사항

- 시험 문제에서 이미 실행 중인 파드를 쫓아내고 싶다는 조건이 붙으면 `NoExecute` 로 설정해야 한다.
- YAML에서 Key, Value 등은 가급적 “”로 묶어주자.
- 반드시 한 쌍(key, value, effect)가 파드의 toleration과 일치해야만 면역력이 발생한다.
