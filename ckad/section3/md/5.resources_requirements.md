# 5. Resource Requirements

이게 왜 생겼을까 생각을 해보자. 하나의 파드 안에 우리는 여러개의 컨테이너를 만들 수 있다. 이를 통해 트래픽 등을 분산하여 처리할 수 있는데, 만약 하나의 파드에 트래픽이 몰려서, 혹은 작업이 몰려서 다른 컨테이너의 작업에 OOM이 뜨는 불상사가 발생할 수 있지 않을까? 따라서 자원을 배분해주기 위해 이러한 것이 생겼다.

#### 파드에 직접 정의

파드에 직접 자원 할당을 지정할 수 있다.

```yaml
apiVersion: v1
kind: Pod
metadata:
	name: web-pod
spec:
	containers:
	- name: web-pod
		image: ubuntu
		ports:
		- containerPort: 8080
		resources:
			requests:
				memory: "4Gi"
				cpu: "2"
```

여기서 `resources` 가 바로 자원 할당이다. cpu의 수는 vCPU의 개수를 의미한다. 1m이 최소이고, 100m은 바로 0.1 vCPU를 의미한다. `requests`는 파드가 이만큼은 최소한 가지고 있어야해요를 나타내는 것이다. 그렇다면 이 Pod의 한계는 이만큼입니다를 지정하는 것은 어떻게 할까?

```yaml
apiVersion: v1
kind: Pod
metadata:
	name: web-pod
spec:
	containers:
	- name: web-pod
		image: ubuntu
		ports:
		- containerPort: 8080
		resources:
			requests:
				memory: "1Gi"
				cpu: "1"
			limits:
				memory: "4Gi"
				cpu: "2"
```

이렇게 된다. 만약 CPU가 한계에 도달하면 throttling이 발생하지만, memory는 OOM을 발생하고 파드가 꺼진다.

---

그런데 우리는 파드를 하나하나 설정해주기 싫어서 configMap과 Secret이 등장한 것을 보았다. 당연히 자원할당에도 비슷한 것들이 존재한다. 먼저 LimitRange이다.

#### LimitRange

```yaml
apiVersion: v1
kind: LimitRange
metadata:
	name: web-limit
spec:
	limits:
	- type: Container
		default: # 리미트 기본값
			cpu: "500m"
		defaultRequest: # 요청 기본값
			cpu: "500m"
		max: # 설정 가능한 최대치
			cpu: "1"
		min: # 설정 가능한 최소치
			cpu: "100m"
```

이거 AWS 자격증 취득할 때도 공부했던 내용들과 비슷하다. 평시에는 500m, 너무 요청이 없는 경우에는 100m, 요청이 폭주할 때는 1로 설정하는 것이다.

네임스페이스 단위로 자원을 할당하는 것도 있다. 이전에 잠깐 봤던 ResourceQuota이다.

#### ResourceQuota

```yaml
apiVersion: v1
kind: ResourceQuota
metadata:
	name: compute-resources
	namespace: dev # 제한이 적용될 네임스페이스의 이름이다.
spec:
	hard:
		requests.cpu: "1"
		requests.memory: 1Gi
		limits.cpu: "2"
		limits.memory: 2Gi
```

이 경우는 cpu, memory 뿐만 아니라 생성할 수 있는 파드, 서비스 수 등을 제한할 수 있다.

```yaml
apiVersion: v1
kind: ResourceQuota
metadata:
	name: nums-limits
	namespace: dev
spec:
	hard:
		pods: "10"
		services: "5"
		services.loadbalancers: "1"
		configmaps: "10"
		secrets: "10"
```

ResourceQuota는 적용 후 반드시 상태를 확인하자.
