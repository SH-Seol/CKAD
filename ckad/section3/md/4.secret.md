# 4. Secret

#### 3. Secret

그래서 보안이 나온 것이다. 비밀을 인코딩하고 아무나 못 보게 하자. 이게 바로 secret의 등장 배경이다.

근데 왜 생긴게 ConfigMap이랑 비슷할까? 얘도 외부 파일이고 다른 곳이 참조해야하는데.. 다르게 생기면 불편하지 않았겠나..

```yaml
# Secret
apiVersion: v1
kind: Secret
metadata:
	name: app-secret
data:
	DB_PASSWORD: #base64

# Pod
apiVersion: v1
kind: Pod
metadata:
	name: app-pod
spec:
	containers:
		- name: my-app
			image: nginx
			env:
				- name: DB_PW
					valueFrom:
						secretKeyRef:
							name: app-secret
							key: DB_PASSWORD
```

이렇게 된 것이다. 그저 외우려고만 하다 외워지지 않으니 이 흐름을 먼저 잡고 가고자 했다.

이제 시크릿을 어떻게 생성할지 한번 알아보자.

```yaml
kubectl create service generic app-secret \
	--from-literal=DB_Host=mysql \
	--from-literal=DB_User=root \
	--from-literal=DB_Password=pw
	
#파일로 부터 생성하고 싶다면
kubectl create service generic app-secret --from-file=./app_secret.properties
```

`--from-literal` 이것은 data에 값을 넣는 방법이라고 한다. 주로 ConfigMap이나 Secret에서만 사용한다고 한다. 기억해야겠지..

데이터를 넣을 때는 반드시 base64 형태로 넣어야한다고 한다.

- 인코딩: `echo -n 'mysql' | base64`
- 디코딩: `echo -n 'bXlzcWw=' | base64 --decode`

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: app-secret
data:
  DB_Host: bXlzcWw=         # 'mysql'의 Base64 값
  DB_User: cm9vdA==         # 'root'의 Base64 값
  DB_Password: cGFzd3Jk      # 'paswrd'의 Base64 값
```

#### 파드에 주입

위에 간단히 봤던 예시는 `valueFrom` 을 사용한 것이니 `envFrom` 의 예시를 적어볼까 한다.

```yaml
apiVersion: v1
kind: Pod
metadata:
	name: myapp-pod
spec:
	containers:
		- name: my-app
		  image: nginx
		  envFrom:
			- secretRef:
			  name: app-secret
```

그런데 파일 형태로 비밀번호를 읽어야하는 경우도 있다고 한다. 이 때는 볼륨으로 마운트한다고 한다. 마운트된 경로로 가면 각 key가 파일의 이름이 된다.

```yaml
spec:
	containers:
		- name: my-app
			volumeMounts:
				- name: secret-volume
					mountPath: "/etc/app-secrets"
					readOnly: true
	volumes:
		- name: secret-volume
			secret:
				secretName: app-secret
```

#### Encryption at Rest

그런데 쿠버네티스의 Secret은 etcd에 평문으로 저장된다. 따라서 etcd에 접근 권한이 있는 사람들은 secret의 암호에 접근할 수 있었다. 그래서 암호화를 더 강화하기 위해 등장한 것이 EncryptionConfiguration이다. 참 웃기긴 하다만, 그냥 Secret을 이렇게 바꾸는 것은 조금 무서웠을지도 모르겠다.

```yaml
apiVersion: apiserver.config.k8s.io/v1
kind: EncryptionConfiguration
resources:
  - resources:
    - secrets
    providers:
	  - aescbc:
		keys:
		  - name: key1
			secret: base64-encoded-secret
```

그래서 이 설정을 `kube-apiserver` 의 옵션 (`--encryption-provider-config` )로 넣어주면 암호화되어 저장된다고 한다.

#### Security Context

SecurityContext는 파드나 컨테이너의 권한과 접근 제어 설정을 정의한다. 컨테이너에게 어떤 userId로 행동할지 어떤 capabilities를 가질지 정해주는 것이다.

Docker와 Kubernetes의 보안 설정 차이를 확인해보자.

- Docker: `docker run —user=1001 —cap-add MAC_ADMIN ubuntu`
- Kubernetes: `spec.securityContext` 또는 `spec.containers[].securityContext` 를 사용

설정 범위에 대해서도 알아보자.

K8s를 보면 spec에서 바로 적는지, 혹은 containers에서 적는 두 방법이 존재한다.

`spec.securityContext` 의 경우는 Pod 수준으로 정의된 것이고, `spec.containers[].securityContext` 의 경우는 container 수준에서 정의된 것이다. 즉 파드 내부에 모두 동일한 보안 설정을 하느냐, 혹은 하나의 컨테이너에만 해당 설정을 해주느냐인 것이고 `capabilities`는 컨테이너 단위에서만 설정이 가능하다. 만약 두 사이에 충돌이 생기면 컨테이너가 우선순위를 가진다.

```yaml
apiVersion: v1
kind: Pod
metadata:
	name: web-pod
spec:
	securityContext:
		runAsUser: 1000
	containers:
		- name: web-container
		  image: ubuntu
		  command: ["sleep", "3600"]
		  securityContext:
			  runAsUser: 2000
		      capabilities:
				add: ["MAC_ADMIN"]
```
