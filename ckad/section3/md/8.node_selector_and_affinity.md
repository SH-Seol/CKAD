파드를 내가 원하는 노드에 넣는 스케줄링 기법에 대해 알아보자.

#### 1. Node Selector

노드에 붙은 Label을 보고 파드가 들어갈 곳을 정하는 것이 바로 Node Selector이다.

- 원리: 노드에 Label을 붙이고 파드에게 이 Label이 붙은 노드로 가라고 하는 원리이다.
- 명령어: `kubectl label nodes <노드명> size=Large`

YAML 예시

```yaml
apiVersion: v1
kind: Pod
spec:
	containers:
	- name: data-processor
		image: data-processor
	nodeSelector:
		size: Large # 이 라벨이 있는 노드만 찾아감
```

**한계:** Large이거나 Medium인 곳같은 OR, NOT은 쓸 수 없다. 딱 맞는 것만 찾아간다.

---

#### 2. Node Affinity

Node Selector의 업그레이드 버전이다. `In` , `NotIn` , `Exists` 같은 연산자를 써서 더 복잡한 조건 등을 걸 수 있다.

- 연산자
    - `In` : 리스트 중 하나라도 일치하면 된다. (OR)
    - `NotIn` : 리스트 내에 일치하지 않으면 된다. (NOT)
    - `Exists` : 값과 상관없이 라벨 키만 있으면 된다

YAML 예시

```yaml
apiVersion: v1
kind: Pod
metadata:
	name: my-pod
spec:
	affinity:
		nodeAffinity:
			requiredDuringSchedulingIgnoredDuringExecution:
				nodeSelectorTerms:
				- matchExpressions:
					- key: size
						operator: In
						values:
						- Large
						- Medium
```

---

#### 3. Node Affinity의 타입

스케줄링할 때 얼마나 엄격하게 지킬 것인지는 정한다.

| 타입 이름 | 스케줄링 시점 | 실행 시점 |
| --- | --- | --- |
| `required..` | 조건 안맞으면 파드 생성 안됨 | 이미 실행 중이면 라벨 바뀌어도 무시 |
| `preferred..`  | 웬만하면 맞추지만 자리 없으면 다른 곳에서 생성됨 | 이미 실행 중이면 라벨 바뀌어도 무시 |

---

#### 4. Taints/Tolerations vs Node Affinity

하나만 써서는 독점 노드를 만들 수 없다. 둘 다 써야지만 정확히 내가 원하는 노드에 파드를 배치할 수 있다.

1. Taints & Tolerations
    - 효과: 원치 않는 파드가 노드에 오는 것을 막음
    - 한계: Toleration이 있는 파드가 반드시 그 노드로 간다는 보장이 없음
2. Node Affinity
    - 효과: 파드를 특정 노드로 끌어당기
    - 한계: 다른 일반 파드가 그 노드에 들어오는 것을 막지 못함

→ Taint + Affinity

1. 노드에 Taint를 뿌려 다른 파드들이 못 오게 막는다
2. 우리 파드에 Toleration을 줘서 그 노드에 접근할 수 있게 한다.
3. 우리 파드에 Node Affinity를 추가하여 전용 노드로 갈 수 있게 한다.
