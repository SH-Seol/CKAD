account는 두개로 나뉜다. 하나는 user account, 다른 하나는 service account.
user account는 개발자와 같은 사람들이 작업을 위해 클러스터에 접근하기 위한 계정이라면 service account는 prometheus나 jenkins와 같이 서비스들이 클러스터에 접근하기 위한 계정이다. 그리고 나는 service account 사용법에 대해 공부하고자 한다.

---

#### 서비스 계정 생성 및 토큰 확인

**생성 명령어**

```yaml
kubectl create serviceaccount mysa
```

**token 생성**

이전에는 계정 생성 시 토큰이 담긴 Secret 오브젝트가 자동으로 생성되었지만 현재는 그렇지 않기에 필요할 때마다 유효기간이 있는 토큰을 직접 생성해야한다. 아래의 명령을 실행하면 화면에 토큰이 출력된다.

```yaml
kubectl create token mysa
```

---

#### 파드와 서비스 계정 연결

**자동 마운트**

따로 설정하지 않으면 모든 파드는 해당 네임스페이스의 default 서비스 계정 토큰을 받는다.

- 위치: `/var/run/secrets/kubernetes.io/serviceaccount/token`
- 앱은 이 파일을 읽어서 API 서버에 Autorization: Bearer 토큰 헤더를 보낸다.

**사용자 지정 서비스 계정 적용**

특정 권한이 있는 계정을 사용하고 싶다면 pod에 이렇게 명시해주면 된다. 다만 이미 실행 중인 파드는 삭제하고 재생성 해야한다.

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: new-sa-pod
spec:
  containers:
    - name: new-sa-pod
      image: ubuntu
  serviceAccountName: mysa # 사용할 service account 이름
  # 쿠버네티스는 자동으로 기본 서비스 계정을 마운트한다. 아래를 false로 하면 그러지 않는다.
  automountServiceAccountToken: false
```

#### 버전별 변경사항

| **구분** | **v1.22 이전** | **v1.22 ~ v1.24 이후** |
| --- | --- | --- |
| **토큰 형태** | 비만료 JWT (Secret 저장) | **만료 시간이 있는** 바인딩된 토큰 |
| **생성 방식** | 서비스 계정 생성 시 자동 Secret 생성 | **TokenRequestAPI**를 통한 동적 생성 |
| **보안성** | 낮음 (유출 시 영구적 위험) | **높음** (시간/대상 제한 가능) |
| **마운트 방식** | Secret 볼륨 마운트 | **Projected 볼륨** 마운트 |

---

**요약**

1. `kubectl create sa 이름` 으로 계정 생성 후 `kubectl create token 이름` 으로 일시적인 토큰 발급받음
2. 추 후에 RBAC로 이 계정에 실제 권한을 연결해줘야함
3. 만약 구형 앱이 반드시 Secret 형태의 토큰을 요구하면 `kubernetes.io/service-account-token` 타입의 secret 직접 만들어 연결
4. 앱이 API를 호출할 일이 없다면 `automountServiceAccountToken: false` 로 설정

#### Lab에서 얻어갈 내용

`kubectl create token dashboard-sa` 를 해서 이미 API access 권한이 있는 dashboard-sa로 deployment의 sa를 변경하고자 한다면

deploy 내의 pod를 의미하는 spec에 spec.serviceAccounName: dashboard-sa를 추가한다.

**1.22+ 버전에서 권장하는 기간 제한 토큰 예제**

1. ServiceAccount 설정 변경

   서비스 계정 자체가 토큰을 자동으로 뿌리지 않도록 잠금 설정

    - `kubectl edit sa dashboard-sa`  or `kubectl patch sa dashboard-sa -p '{"automountServiceAccountToken": false}'`
    - 핵심은 automountServiceAccountToken: false를 추가하는 것
    - 보안상의 이유로 모든 파드에 토큰이 자동으로 들어가는 것을 방지하기 위함.
2. Deployment 수정

   deployment가 필요한 토큰을 직접 가져오도록 설계 수정

    1. 파드 수준 설정(`spec.template.spec`)
        - `serviceAccountName: dashboard-sa` 지정
        - `automountServiceAccountToken: false` 확실하게 한번 더 기입
    2. 볼륨 정의(`volumes`)

       Projected Volume이라는 특수한 볼륨 생성

        - 이름: `token`
        - 소스: `serviceAccountToken`
        - 경로: `path: token`
    3. 컨테이너에 마운트(`volumeMounts`)
        - 경로: `/var/run/secrets/kubernetes.io/serviceaccount/`
        - 읽기 전용: `readOnly: true` (컨테이너가 토큰을 수정할 수 없게 방어)

    ---

   #### 최종 YAML 구조

    ```yaml
    spec:
    	template:
    		spec:
    			containers:
    			- name: web-dashboard
    				volumeMounts:
    				# 아래 위치는 쿠버네티스 표준 토큰 위치이므로 관례를 따르는 것이 트러블슈팅에 유리
    				- mountPath: /var/run/secrets/kubernetes.io/serviceaccount/
    					name: token
    					readOnly: true
    			serviceAccountName: dashboard-sa
    			volumes:
    			- name: token
    				projected:
    					sources:
    					- serviceAccountToken:
    							path: token
    ```
