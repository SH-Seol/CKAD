# 6. Namespace

#### 1. Namespace란?

쉽게 생각하면 집이다. 물리적인 클러스터안에서 논리적으로 격리된 공간이다.

- 집 내부: 같은 namespace 안에서는 서비스 이름만으로 통신이 가능하다.
- 집 외부: 다른 namespace에 접근하기 위해서는 전체 이름(fullname)인 FQDN을 사용해야한다.

---

#### 2. 기본 생성 namespace

쿠버네티스를 설치하면 자동으로 생기는 namespace들이 존재한다.

- default: 아무 설정 안했을 때 리소스가 생성되는 기본 장소
- kube-system: 네트워킹, DNS 등 쿠버네티스 시스템 자체를 위한 컴포넌트들이 있는 곳
- kube-public: 모든 사용자에게 공개되어야 하는 정보가 담기는 곳

---

#### 3. 네임스페이스 간 통신(DNS)

다른 네임스페이스에 있는 DB-Service에 접속하고 싶으면 아래의 형식을 따른다.

형식: `[servicename].[namespace].svc.cluster.local`

ex: `db-service.dev.svc.cluster.local`

---

#### 4. YAML 코드

먼저 생성을 위한 기본 명령어는 다음과 같다.

```yaml
kubectl create namespace dev
```

그리고 yaml파일은 아래와 같이 작성하면 된다.

```yaml
apiVersion: v1
kind: Namespace
metadata:
	name: dev
```

리소스 조회 및 생성은 다음과 같다.

```yaml
#모든 네임스페이스의 파드들을 조회
kubectl get pods --all-namepsace

#특정 이름을 가진 네임스페이스의 파드를 조회 여기서는 dev
kubectl get pods -n dev

#특정 네임스페이스에 파드를 생성
kubectl apply -f pod.yaml --namespace=dev
kubectl apply -f pod.yaml -n dev
```

또한 작업을 할 때 계속 default에서 다른 네임스페이스에 작업하겠다고 추가적으로 붙이는 것보다 기본 위치를 수정할 수 있다.

```yaml
kubectl config set-context $(kubectl config current-context) --namespace=dev
```

---

#### 5. 리소스 할당

특정 네임스페이스에서 너무 과한 리소스 사용을 막기 위해서 리소스 할당을 할 수 있다. 여기서 사용하는 것이 `ResourceQuota` 이다.

```yaml
apiVersion: v1
kind: ResourceQuota
metadata:
	name: compute-quota
	namespace: dev
spec:
	hard:
		pods: "10"
		requests.cpu: "4"
		requests.memory: 5Gi
		limits:cpu: "10"
		limits:memory: 10Gi
```

이를 생성하는 명령어도 다른 것들과 같다.

```yaml
kubectl create -f compute-quota.yaml
```

#### 주의사항

1. `--namespace` or `-n` 옵션을 반드시 확인하자. 특정 네임스페이스에 파드를 생성해야할 때 이를 반드시 활용해야한다.
2. FQDN을 기억하자. 다른 네임스페이스의 서비스를 호출할 때 주소 체계 `svc.cluster.local` 를 기억하고 네트워크 트러블슈팅 문제를 풀자.
3. 네임스페이스는 논리적 격리일 뿐, network policy가 없다면 기본적으로 서로 소통이 가능하다.
