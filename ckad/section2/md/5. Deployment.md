# 5. Deployments

Deployment는 Pod와 ReplicaSet이라는 기초공사 위에 세워지는 관리 시스템이라고 생각하면 편하다.

#### 1. 왜 Deployment인가?

레플리카셋은 지정된 개수의 파드를 유지하는 것에 집중하지만, Deployment는 애플리케이션의 생명주기 전체를 관리한다.

- 롤링 업데이트(Rolling Update): 서비스 중단 없이 파드를 하나씩 최신 버전으로 교체한다.
- 롤백(Rollback): 업데이트 중 에러가 발생하면 즉시 이전의 안정적인 버전으로 돌아간다.
- 상태 관리: 일시 중지, 재개, 스케일링 등 복합적인 변경사항을 조율한다.

---

#### 2. 계층 구조

쿠버네티스 오브젝트는 층층이 쌓여 있다. 이 관계를 이해하는 것이 deployment의 핵심이다.

1. Pod: 실제 컨테이너가 돌아가는 최소 단위
2. ReplicaSet: Deployment의 지시를 받아 지정된 수의 파드를 유지
3. Deployment: 어떻게 업데이트할지 전략을 결정하고 ReplicaSet을 관리

> 우리가 Deployment를 업데이트하면 Deployment는 새로운 ReplicaSet을 만들고 기존 ReplicaSet의 Pods를 하나씩 옮겨온다.
>

---

#### 3. YAML 코드

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
	name: myapp-deployment
	labels:
		app: myapp

spec: 
	replicas: 3
	selector:
		matchLabels:
			type: front-end
	template:
		metadata:
			name: myapp-pod
			labels:
				app: myapp
				type: front-end
		spec:
			containers:
				- name: nginx-container
					image: nginx
```

ReplicaSet과 거의 동일한데 kind만 `Deployment`로 수정했다.

---

#### 4. 필수 명령어

| 작업 | 명령어 | 설명 |
| --- | --- | --- |
| 생성 | `kubectl create -f deployment-definition.yml` | 파일을 통해 생성 |
| 생성(명령형) | `kubectl create deployment --image=nginx` | 파일 없이 생성 |
| 조회 | `kubectl get deployments` | deployment 목록 조회 |
| 전체 확인 | `kubectl get all` | deployment, replicaset, pod 전부 조회 |
| 상세 분석 | `kubectl describe deployment myapp-deployment` | 업데이트 진행 상황 등 확인 |

---

**롤링 업데이트 시각화**

새로운 이미지 버전으로 업데이트할 때 Deployment는 한번에 업데이트하지 않고 점진적으로 업데이트해서 가용성을 유지한다.

#### 주의사항

- 수정은 Deployment에서 하자. Deployment를 수정해야 위계에 맞춰 replicaset, pod가 자동으로 수정된다.
