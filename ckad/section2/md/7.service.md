서비스 파트에서는 TargetPort, Port, NodePort의 차이를 정확히 이해하는 것이 중요하다.

#### 1. Service란?

쿠버네티스에서 파드는 언제든 죽고 다시 살아날 수 있으며, 그럴 때마다 IP가 계속 바뀐다. 서비스는 이 파드들에게 고정적인 입구를 제공한다.

- 느슨한 결합: 프론트엔드가 백엔드 파드IP가 바뀐 것을 알 필요가 없이 서비스를 통해 소통하면 된다.
- 통신 지원: 사용자 ↔ 프론트엔드, 프론트엔드 ↔ 백엔드 연결을 모두 담당한다.

---

#### 2. Service의 3가지 주요 유형

| 유형 | 통신 범위 | 주요 용도 | 특징 |
| --- | --- | --- | --- |
| ClusterIP | 클러스터 내부 | 백엔드 ↔ DB, 프론트 ↔ 백엔드 | 서비스의 기본 타입. 내부 전용 가상 IP 생성 |
| NodePort | 클러스터 외부 | 외부 사용자 ↔ 프론트엔드 | 모든 노드의 특정 port(30000 ~ 32767) 개방 |
| LoadBalancer | 클러스터 외부 | 실제 운영 환경 서비스 노출 | 클라우드 공급자(AWS, GCP 등)의 로드밸런서와 연결 |

---

#### 3. 서비스의 3가지 포트

NodePort 서비스를 설정할 때 다음 세 가지 포트를 설정해야한다. 서비스를 기준으로 기억하자.

1. NodePort: 외부 사용자가 노드의 IP로 접속할 때 사용하는 포트 (30000~32767)
2. Port(서비스 포트번호): 서비스 오브젝트 자체의 포트 (클러스터 내부에서 서비스 IP로 접근할 때 사용)
3. TargetPort(서비스가 목표로 하는 파드): 실제 애플리케이션 파드가 리스닝하고 있는 포트 번호

---

#### 4. YAML 코드

먼저 ClusterIP를 알아보자. `spec.type` 을 작성하지 않으면 기본적으로 clusterIP가 생성된다.

ClusterIP는 내부 통신용이므로 NodePort번호를 설정하지 않아도 된다.

```yaml
apiVersion: v1
kind: Service
metadata:
	name: backend-service
spec:
	type: ClusterIP
	ports:
		- port: 80
			targetPort: 80
	selector:
		app: myapp
		type: back-end
```

NodePort는 외부 노출용이다.

```yaml
apiVersion: v1
kind: Service
metadata:
	name: myapp-service
spec:
	type: NodePort
	ports:
		- port: 80
			targetPort: 80
			NodePort: 30008
	selector:
		app: myapp
		type: front-end
```

ports에서 무조건 들어가야하는 것은 `port` 뿐이다.

#### 주의할 점

1. 레이블/셀렉터 일치: `selector` 는 반드시 대상 파드의 `labels` 와 100% 일치
2. DNS 활용: 클러스터 내부에서는 IP 대신 서비스 이름을 사용(ex. `mysql.connect("db-service")` )
3. 자동 업데이트: 파드가 새로 생기거나 삭제되어도 서비스의 `selector` 가 레이블을 자동으로 감시하여 엔드포인트를 업데이트하므로 별도의 설정 변경 필요 x
