우리가 파드를 생성할 때 그 안에 컨테이너를 하나만 넣을 수 있는 것이 아니다. 여러개를 넣을 수 있다. 이번에는 그것에 대해 정리해보겠다.

#### 1. Multi Container Pod란?

하나의 파드 안에 두 개 이상의 컨테이너를 함께 실행하는 방식이다. 마이크로서비스의 철학을 유지하면서도 강하게 결합되어야하는 서비스들을 위해 사용된다.

**멀티 컨테이너의 3대 공유 자원**

1. 공통의 lifecycle: 파드가 생성될 때 함께 생성되고 삭제될 때 함께 삭제
2. 네트워크 공유: 모든 컨테이너가 동일한 네임스페이스를 공유하므로 서로를 `localhost`로 호출 가능
3. 스토리지 공유: 동일한 볼륨에 접근할 수 있어 데이터 공유가 매우 쉬움

---

#### 2. 멀티 컨테이너 파드 생성(YAML)

```yaml
apiVersion: v1
kind: Pod
metadata:
	name: myapp-pod
spec:
	containers:
	- name: main-app
		image: main-app
		ports:
		- containerPort: 8080
	- name: sidecar
		image: log-agent
```

`spec.containers` 가 list로 인자를 받음을 확인하자.

---

#### 3. 멀티 컨테이너 디자인 패턴

| 패턴 | 주요 역할 | 예시 |
| --- | --- | --- |
| Sidecar | 메인 컨테이너의 기능을 확장/보조 | 로그 수집기, 성능 모니터링 에이전트 |
| Adapter | 메인 컨테이너의 출력을 표준화/변환 | 서로 다른 형식의 로그를 중앙 시스템용 포맷으로 변환 |
| Ambassador | 외부 서비스(DB 등)와의 통신 프록시 | 환경(Dev/Prod)에 따른 DB 라우팅 처리 |

---

#### 4. 실행 순서에 따른 분류(Init vs Sidecar)

시작 순서와 수명 주기에 따라 차이가 존재한다.

1. Co-located 컨테이너
    - 특징: `spec.containers` 배열에 나열된 순서대로 시작된다는 보장이 없음
    - 용도: 대등한 관계의 서비스들
2. Init 컨테이너
    - 특징: 메인 앱이 뜨기 전 실행을 완료하고 종료되어야 함. 여러 개일 경우 순차 실행
    - 용도: DB 준비 상태 확인, 설정 파일 미리 생성 등
3. Sidecar 컨테이너
    - 특징: 메인 앱보다 먼저 시작되어야 하지만, Init container와 달리 종료되지 않고 파드 수명 내내 함께 실행
    - 용도: 로그 에이전트 등

---

#### 주의사항

1. YAML에서 containers는 list이다.
2. 멀티 컨테이너끼리는 별도의 서비스 없이 `127.0.0.1` 로 통신이 가능하다.
3. `kubectl logs <pod>` 만 치면 에러가 난다. 컨테이너가 여러 개일 때는 반드시 `kubectl logs <pod> -c <container>` 명령어를 통해 지정해줘야 한다.
